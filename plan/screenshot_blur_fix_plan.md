# 截图清晰度修复与链路审查方案

## 1. 任务目标

- 基于当前仓库代码，分析“截图后图片模糊”的真实原因。
- 区分“默认链路缺陷”和“用户配置导致的二次质量损失”，避免误判。
- 给出可落地的修复路径，并明确各方案适用范围与风险。

本计划基于当前代码静态分析整理，重点核对了以下文件：

- `src/core/capture/screen_capture_service.cpp`
- `src/core/capture/screen_capture_service.h`
- `src/core/capture/region_selection_dialog.cpp`
- `src/core/capture/annotation_dialog.cpp`
- `src/app/shell/bridge/capture_controller.cpp`
- `src/app/shell/qml/pages/CapturePage.qml`
- `src/core/settings/settings_manager.cpp`

## 2. 现状结论

从当前代码看，**默认截图发糊的头号嫌疑，是虚拟桌面拼接阶段把高 DPI 原始截图压回了逻辑坐标尺寸**，而不是 PNG 保存格式本身。

但文档里原先把“输出倍率”也放到同级 P0，有些不够严谨。因为：

- 默认配置是 `100%`，这条路径本身不会触发二次缩放；
- 只有用户把倍率改到非 `100%` 时，`scaleImage()` 才会额外做一次插值缩放；
- 所以它是**高频的质量风险项**，但不应直接等同于“默认就模糊”的主因。

另外，当前文档对“多屏问题”的表述也偏窄。实际上这条问题链在**单屏高 DPI** 下同样成立，只是混合 DPI 多屏场景更容易放大问题。

## 3. 根因分析修正版

### 3.1 根因 A（P0，主因）：虚拟桌面拼接使用逻辑坐标画布，导致高 DPI 原图被隐式缩小

证据：

- `captureVirtualDesktop()` 先用 `screen->geometry()` 合并出 `virtualRect`，这是逻辑坐标空间。
- 然后直接用 `virtualRect.size()` 创建 `stitchedImage`。
- 最后把 `screen->grabWindow(0)` 得到的 `QPixmap` 直接画到这张逻辑尺寸画布上。

对应代码：

- `src/core/capture/screen_capture_service.cpp:249-261`
- `src/core/capture/screen_capture_service.cpp:270-277`

为什么这是主因：

1. `screen->grabWindow(0)` 在高 DPI 屏上拿到的底层像素，通常比逻辑尺寸更大。
2. 当前目标画布却只有逻辑尺寸。
3. `QPainter::drawPixmap()` 在这里会把带高 DPI 信息的截图按逻辑尺寸落到普通 `QImage` 画布上，结果就是隐式重采样。
4. 一旦发生这一步，后面无论区域裁剪还是标注，拿到的都已经是“缩水后的底图”。

结论：

- 这不是“仅多屏拼接”问题，而是**整个虚拟桌面截图链路的坐标空间选错了主存储格式**。
- 多屏、尤其混合 DPI 只是让问题更明显，不是问题成立的前提。

### 3.2 根因 B（P1，条件性风险）：输出倍率会对结果做二次插值缩放

证据：

- 保存前统一经过 `saveImage()` -> `scaleImage()`。
- 当 `scalePercent != 100` 时，会调用 `source.scaled(..., Qt::SmoothTransformation)`。
- UI 允许用户把输出倍率调整到 `50%~200%`，并且该配置会持久化。

对应代码：

- `src/core/capture/screen_capture_service.cpp:308-352`
- `src/app/shell/qml/pages/CapturePage.qml:206-223`
- `src/core/settings/settings_manager.cpp:278-285`
- `src/core/settings/settings_manager.cpp:500`
- `src/core/settings/settings_manager.cpp:551-554`

结论：

- 这是**确定会影响清晰度**的逻辑。
- 但它的前提是用户把倍率调离 `100%`。
- 所以更准确的表述应是：
  - 它不是“默认截图模糊”的第一主因；
  - 但它会显著增加“用户截图发糊”的反馈概率，值得优先治理。

### 3.3 根因 C（P1，继发问题）：区域截图与导出裁剪强绑定 `snapshot.image`，会继承底图失真

证据：

- 区域选择对话框使用 `QPixmap::fromImage(snapshot.image)` 作为底图。
- 最终裁剪直接对 `snapshot.image` 执行 `copy(region)`。
- `region` 也直接与 `snapshot.image.rect()` 相交，没有独立的逻辑坐标到导出像素坐标映射。

对应代码：

- `src/core/capture/screen_capture_service.cpp:134-158`

结论：

- 这条链路会**完整继承根因 A 的损失**。
- 但它本身不是新增的一次模糊来源，而是把已经模糊的底图继续传下去。
- 更重要的是：一旦我们修复根因 A、让导出图不再等于逻辑尺寸，这里的实现就会立刻不够用，因此它既是当前问题的放大器，也是后续修复的关键阻塞点。

### 3.4 因素 D（P2，感知层问题）：选择框和标注窗口的预览缩放会让“看起来模糊”更明显

证据：

- 区域选择窗口每次 `paintEvent()` 都把整张截图绘制到窗口 `rect()`。
- 标注窗口同样通过 `imageRect()` 把图像缩放到当前可用显示区域。
- 鼠标坐标再通过比例换算映射回图像坐标。

对应代码：

- `src/core/capture/region_selection_dialog.cpp:37-65`
- `src/core/capture/annotation_dialog.cpp:171-174`
- `src/core/capture/annotation_dialog.cpp:364-409`

结论：

- 这更偏向**预览层的视觉模糊**，不直接决定最终导出 PNG 是否清晰。
- 但如果底图已经被压缩，这两个窗口会进一步强化用户“截图很糊”的主观感受。

## 4. 当前计划需要修正的地方

### 4.1 “输出倍率”不宜继续和主因并列为默认 P0

建议改为：

- 根因 A：P0 主因
- 根因 B：P1 条件性高风险项
- 根因 C：P1 继发问题 / 修复阻塞点

这样优先级更符合真实代码行为。

### 4.2 “多屏截图拼接”表述过窄，应扩大为“虚拟桌面截图链路”

建议把文中的：

- “多屏截图拼接时未正确处理高 DPI / devicePixelRatio”

改成：

- “虚拟桌面截图链路在高 DPI 下使用了逻辑坐标画布，导致原始截图被隐式缩小；混合 DPI 多屏场景会进一步放大问题”

这样不会误导成“单屏不受影响”。

### 4.3 方案一的收益边界需要写清楚

原计划里“方案一：最小改动优先”写得偏乐观，尤其是这句目标：

- “双屏混合缩放场景下，不同屏幕截图清晰度保持一致”

这条在当前架构下不应轻易承诺。原因是：

- 单张导出图如果既要保留每块屏幕原生像素，又要继续用统一逻辑坐标交互，就需要额外的映射数据；
- 对混合 DPI 多屏来说，这已经不只是改一个 `captureVirtualDesktop()` 那么简单；
- 如果验收标准是“混合 DPI 多屏也必须按各屏原生像素无损导出”，那更接近架构级修复，而不是小修。

建议把方案边界改成：

- 方案一可优先解决：单屏高 DPI、同 DPI 多屏、输出倍率误用；
- 若产品验收要求覆盖：混合 DPI 多屏原生精度导出，则应直接转向方案二或至少引入映射模型。

### 4.4 需要先补运行时取证，再进入改代码阶段

当前计划几乎全靠静态分析，方向基本对，但还缺一组非常关键的运行时证据：

- 每块屏幕的 `geometry`
- 每块屏幕的 `devicePixelRatio`
- `grabWindow(0)` 返回图像的像素尺寸
- `grabWindow(0)` 的 `devicePixelRatio`
- `stitchedImage.size()`
- 最终保存图尺寸
- 当前 `scalePercent`

没有这组数据，后面即使改对了，也不容易证明“到底是高 DPI 画布问题修好了，还是只是碰巧看起来好一点”。

## 5. 修正后的方案构思

### 方案一：分阶段保守修复（推荐）

目标：

- 先修复默认链路中最确定的清晰度损失；
- 控制改动范围；
- 不对混合 DPI 多屏的“原生像素无损”做过度承诺。

核心思路：

1. 先补日志和运行时取证，确认当前各屏逻辑尺寸、物理截图尺寸与 DPR 的关系。
2. 修复 `captureVirtualDesktop()` 的高 DPI 处理，优先覆盖单屏高 DPI 和同 DPI 多屏。
3. 将“输出倍率”从普通设置改成带风险提示的高级选项。
4. 在后续阶段再补区域裁剪与标注的坐标映射改造。

优点：

- 改动路径更稳，适合先解决用户最直观问题。
- 能快速甄别“底图模糊”与“二次缩放模糊”。
- 适合先落地一轮 P0。

缺点：

- 若要完整覆盖混合 DPI 多屏原生精度导出，后续仍可能需要扩展为方案二。

适用条件：

- 当前主要用户反馈集中在单屏或统一缩放场景；
- 希望先尽快恢复可用性，再决定是否做架构升级。

### 方案二：引入逻辑坐标与导出坐标映射模型

目标：

- 从模型层解决高 DPI、混合缩放、多屏拼接、区域裁剪、标注导出一致性问题。

核心思路：

1. `DesktopSnapshot` 不再只保存一张 `image`，而是同时保存：
   - 逻辑空间的预览信息；
   - 每块屏幕的原始截图；
   - 逻辑坐标到导出像素坐标的映射信息。
2. 区域选择和标注继续在逻辑空间交互。
3. 最终导出时根据映射关系，从原始截图中裁剪并合成最终图像。

优点：

- 能从根上处理混合 DPI 多屏。
- 后续扩展窗口截图、滚动截图等能力时更稳定。

缺点：

- 开发和验证成本明显更高。
- 需要同步调整区域选择、标注、导出、日志与测试。

适用条件：

- 产品验收明确要求混合 DPI 多屏也保持原生精度；
- 或者后续截图能力还会继续扩展，不适合重复返工。

### 推荐结论

推荐先执行**方案一**，但要把它的边界写清楚：

- 第一轮修复目标是“默认截图不再明显发糊”；
- 如果在 mixed-DPI 多屏验收中仍发现无法满足“原生精度”要求，再升级到方案二；
- 不建议在文档里直接承诺“最小改动即可完整解决混合 DPI 多屏原生清晰度问题”。

## 6. 修正后的实施计划

### 阶段 0：先补取证日志（P0）

目标：让问题诊断从“主观观感”变成“有数据可复现”。

建议记录：

- 当前截图模式（全屏 / 区域）
- 屏幕数量
- 每块屏幕：
  - `geometry()`
  - `devicePixelRatio()`
  - `grabWindow(0)` 返回尺寸
  - `grabWindow(0)` 的 DPR
- `stitchedImage` 尺寸
- `scalePercent`
- 最终保存尺寸

建议落点：

- `src/core/capture/screen_capture_service.cpp`
- `src/app/shell/bridge/capture_controller.cpp`

### 阶段 1：修复虚拟桌面底图清晰度（P0）

目标：解决默认路径的主要模糊来源。

改动重点：

1. 重新定义 `captureVirtualDesktop()` 的输出语义。
   - 不再默认认为“逻辑桌面尺寸 == 导出图像尺寸”。
   - 至少要明确区分“交互坐标空间”和“导出像素空间”。

2. 优先修复单屏高 DPI / 同 DPI 多屏。
   - 保证抓下来的原始图不会在拼接阶段被提前压缩。

3. 若阶段 1 仍坚持只返回单张 `QImage`，则必须同步评估其对混合 DPI 多屏的边界。
   - 不能在文档里默认认为这个结构天然足够。

交付结果建议改为：

- 单屏 100% / 125% / 150% / 200% 下，全屏截图不再明显发虚；
- 同 DPI 多屏场景，全屏截图保持清晰；
- 混合 DPI 多屏场景，先验证结果，再决定是否升级为方案二。

### 阶段 2：治理输出倍率误用（P1）

目标：消除“二次缩放导致模糊”的高频反馈来源。

改动重点：

1. UI 提示明确化。
   - 在“输出倍率”旁增加说明：非 `100%` 会改变像素尺寸并可能影响清晰度。

2. 保存路径可观测化。
   - `scalePercent == 100` 时走直存路径；
   - 非 `100%` 时写日志，便于排查。

3. 视产品定位决定是否收紧能力。
   - 如果产品强调“原样截图”，建议只保留 `100%`；
   - 如果保留缩放功能，应标记为高级项，而不是默认常规项。

### 阶段 3：修复区域裁剪与标注映射（P1）

目标：让交互层继续使用逻辑坐标，同时保证导出结果落在正确像素位置。

改动重点：

1. 重新定义 `RegionSelectionDialog::selectedRegion()` 的语义。
   - 它当前返回的是窗口交互坐标；
   - 如果导出图不再与窗口一一等尺寸，需要显式做映射。

2. 区域裁剪不再直接依赖 `snapshot.image.copy(region)`。
   - 应改为先把逻辑选区转换到导出坐标，再执行裁剪。

3. 复核标注窗口的坐标映射。
   - 当前 `widgetToImage()` 是按显示矩形比例映射到图像像素；
   - 当底图语义变化后，这套逻辑可能要跟着更新。

### 阶段 4：补异常处理与回归测试（P1）

目标：保证问题修复后可持续维护。

建议补充：

- 单屏抓取失败时的明确日志，而不是直接 `continue`
- 保存失败、缩放失败、空图像等错误上下文
- 至少一组图像尺寸/坐标映射的自动化测试或回归脚本

## 7. 其他问题审查结论

除“模糊”外，原计划提到的以下问题仍然成立，可以保留：

### 7.1 高优先级

证据：

- `src/core/capture/annotation_dialog.cpp:975-1013`

- 高分辨率截图进入标注后，多次撤销会显著增加内存占用。

## 8. 验证方案修正版

### 8.1 基础验证

1. 单屏 100% / 125% / 150% / 200% 下执行全屏截图。
2. 同 DPI 双屏场景执行全屏截图。
3. mixed-DPI 双屏场景执行全屏截图。
4. 区域截图选择细小文字、1px 边框、图标边缘做对比。

### 8.2 质量验证

1. 对比系统自带截图工具相同区域结果。
2. 重点观察：
   - 字体边缘是否发虚；
   - 图标边缘是否出现明显平滑；
   - 各屏拼接处是否出现清晰度变化；
   - 区域截图是否比整图裁出来更糊。

### 8.3 数据验证

1. 核对运行日志中每块屏幕的逻辑尺寸、DPR 与抓图尺寸是否匹配。
2. 核对最终导出尺寸是否符合预期。
3. 核对当 `scalePercent != 100` 时，日志是否明确标明发生了二次缩放。

## 9. 最终结论

这份计划**可以完善，而且值得修改**。需要修正的核心点有三条：

1. 把“虚拟桌面高 DPI 拼接问题”明确为默认主因。
2. 把“输出倍率”降级为条件性高风险项，而不是默认主因。
3. 把“方案一”的能力边界写清楚，尤其不要对 mixed-DPI 多屏原生精度做过度承诺。

按当前代码事实，更准确的优先级应该是：

1. **P0：虚拟桌面截图链路在高 DPI 下使用逻辑尺寸画布，导致底图被隐式缩小**
2. **P1：输出倍率会在非 100% 时引入二次插值缩放**
3. **P1：区域截图与标注链路继承底图质量问题，并在后续修复中需要补坐标映射**

如果后续进入编码阶段，建议先按“阶段 0 -> 阶段 1 -> 阶段 2”推进；只有当 mixed-DPI 多屏验收仍不满足时，再升级到方案二。
